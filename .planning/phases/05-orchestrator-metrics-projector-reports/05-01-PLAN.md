<objective>
Resolve the two build/render seam gaps that block Phase 5's `run` orchestrator, per **D5-13**, with a **minimal additive refactor of `src/pipeline/runStack.ts`** — no rewrite:

1. **Skip-copy (gap #1):** when the workspace is already agent-populated, do not re-copy the pristine template over the agent's work. Today `runStack` unconditionally calls `copyWorkspace(stack.template, runId, TMP_ROOT)` at the top, which would overlay the skeleton and build the template instead of the agent's app.
2. **Server-up evaluation window (gap #2):** yield the live Playwright `page` (via the existing `renderWithPage`, which returns `{png, page, close}`) to an injected `onLivePage` callback **before** teardown, so `evaluateRun`'s axe evaluator can run against the agent's real running app. Today `runStack` screenshots with the tear-down `createPlaywrightRenderer` and then `killProcessTree`s the server — leaving axe with no live page.
3. **Startup/render telemetry (TEL-03):** emit `stage_started/completed{stage:'start'}` around server readiness and `stage_started/completed{stage:'render'}` around the screenshot, so startup and render durations fold from the event log like `install`/`build` (they are not in the log today).

The fatal-stage→`RunOutcome` mapping and orphan-proof teardown stay in **one place** (runStack), per D5-13 — the orchestrator does not recompose the primitives. This is additive parameterization: all existing 3-arg callers keep working.

Purpose: unblock the Phase 5 orchestrator (CLI-01) and make startup/render foldable (TEL-03) without duplicating teardown/outcome logic.
Output: a parameterized `runStack` (prePopulated + onLivePage), a one-literal `Stage` widening, and extended integration coverage.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent-generated app → build/serve (execa) | Untrusted agent-written files get `npm ci` / `ng build` / `ng serve` run on them in the isolated `tmp/<runId>/angular` workspace. |
| served app → headless Chromium (renderWithPage) | The agent's app runs in a headless browser; the onLivePage window runs eval JS (axe, later) against it. |
| runStack ← orchestrator callback (onLivePage) | A caller-injected async fn executes inside runStack's server-up window before teardown. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Denial of Service | `startServer` subprocess kept alive during the onLivePage window | low | mitigate | `killProcessTree(subprocess)` stays in the outer `finally` and runs on every exit path AFTER the eval window (POSIX process-group kill, WORK-04) — unchanged; no new orphan path. onLivePage is wrapped in `try/finally { await close() }` so the browser closes even if the callback throws. |
| T-05-02 | Tampering | `prePopulated` skips the template copy and builds the agent-mutated dir | low | accept | Building/serving untrusted agent code in the disposable `TMP_ROOT` under `buildAllowlistedEnv` + array-form execa (no shell) is the pre-existing Phase-2 benchmark premise. `prePopulated` changes WHICH files are built, not the isolation posture; `appDir = resolve(TMP_ROOT, runId, "angular")` uses internal constants only (never spec-derived), so isolation holds by construction. |
| T-05-03 | Information Disclosure | live page + injected callback executes against the localhost dev server | low | accept | Headless Chromium with determinism controls (`blockExternalFonts`, `installDeterminismControls`) against `localhost` only; the callback is orchestrator-owned code, not external input. No new network egress. |
| T-05-SC | Tampering (supply chain) | npm/pip/cargo installs | n/a | accept | This plan adds NO package installs — it uses stdlib + already-installed `renderWithPage`/render modules. No `## Package Legitimacy Audit` entry applies; no install checkpoint needed. |

**No new high-severity surface.** This plan only reorders the existing server-up window and widens a type union. Every runtime risk it touches (orphan processes, untrusted-code build, headless render) already has a Phase-2 mitigation that stays in place. ASVS L1, block-on-high: nothing at high → no block.
