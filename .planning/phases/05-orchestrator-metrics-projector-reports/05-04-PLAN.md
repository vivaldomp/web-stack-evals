<objective>
Implement `projectMetrics(db, runId)` in `src/telemetry/projectMetrics.ts`: a pure fold of the persisted append-only event log into the three projection tables `metrics` / `tool_calls` / `iterations`. It reads `readEvents(db, runId)` (already seq-ordered) AFTER the run and writes rows through bound-param prepared SQL — metrics are projections, never computed inline (TEL-02 / D-24). Covers performance (wall/build/ttft/startup/cost/tokens + `backoff_wait_ms`, D5-12), engineering (files created/edited, lines +/−), iteration count + correction density (D5-11), and per-type tool-call counts (TEL-03/04/05/06).

Purpose: a wrong fold silently produces wrong benchmark numbers — this is the product-critical risk of Phase 5. Every fold is proven by a deterministic golden fixture (fixtures ARE the spec) plus determinism / order-invariance / sum-conservation properties.

Output: `src/telemetry/projectMetrics.ts` exporting `projectMetrics(db, runId): void`; Wave-0 tests `tests/projector.test.ts` + `tests/projectionNotInline.test.ts`, all green.
</objective>

  <action>
    Implement and export `projectMetrics(db: Database.Database, runId: string): void`. Read the log once via `readEvents(db, runId)` (already seq-ordered — do not re-sort). Compute all folds in a single pass (plus one grouping pass), then, inside one `db.transaction`, DELETE existing rows for this run_id from metrics/tool_calls/iterations and INSERT the freshly folded rows — delete-then-insert is what makes re-running idempotent (the tables have no UNIQUE key). Define module-level prepared-SQL constants (delete + insert per table) mirroring evaluations.ts; every value is a named bound parameter, never concatenated (@run_id used solely as a bound param).

    Metric folds → `metrics(name,value,unit)`:
    - Performance (unit "ms"): for every stage present in stage_completed/stage_failed events, write `<stage>_ms` = its durationMs (data-driven, so install_ms/build_ms/lint_ms/test_ms and start_ms — startup, D5-13 — and any render duration fold uniformly when emitted; absent stages produce no row). `ttft_ms` = first_token.ts − session_started.ts (omit if either absent). `wall_ms` = benchmark_finished.ts − session_started.ts, falling back to max(ts) − min(ts) over all events when session_started is absent.
    - Cost/tokens: `cost_usd` (unit "usd") = Σ usage.costUsd across ALL usage events including aborted ones (D4-15) — verbatim, unrounded (D-26). `input_tokens`/`output_tokens`/`cache_read_tokens`/`cache_write_tokens`/`total_tokens` (unit "tokens") = Σ of the respective usage fields.
    - `backoff_wait_ms` (unit "ms", D5-12): scan `unknown` events in seq order; recognize retry-start / retry-end by piType (fixture uses 'auto_retry_start' / 'auto_retry_end'); pair each start with the next matching end and sum (end.ts − start.ts). Timestamp-derived so it survives Pi field renames (A2); pairs by seq order.
    - Engineering (unit "count"): `files_created` = number of distinct paths having ≥1 file_mutation with op='create'; `files_edited` = distinct paths having ≥1 op='edit'; `lines_added` = Σ file_mutation.linesAdded; `lines_removed` = Σ file_mutation.linesRemoved (best-effort — Pi often reports 0, which is a valid value, not missing data).
    - Iteration (unit "count"/"ratio"): `iteration_count` = number of usage events (one per turn_end, D4-11). `correction_density` (unit "ratio") = totalCorrections / iteration_count, guarded to 0 when iteration_count = 0.

    Correction + iterations fold (D5-11, seq-ordered): single walk over the seq-ordered events maintaining `iterationIndex` (starts at 0, incremented on each usage event = count of usage seen so far) and `Map<path, writeCount>`. For each file_mutation: increment writeCount[path]; if writeCount[path] > 1 it is a correction → increment the correction tally for `min(iterationIndex, iteration_count − 1)` (clamp so a correction after the last turn still lands on the final row and Σ is conserved). After the walk, write one `iterations(iteration_index, correction_count)` row per turn for iteration_index 0..iteration_count−1 (correction_count defaults 0 for turns with none); when iteration_count = 0 write no rows. Σ over these rows equals the standalone total-corrections count and equals correction_density × iteration_count (the sum-conservation property).

    Tool-call fold → `tool_calls(tool_name,call_count,error_count)` (TEL-06): group tool_call events by toolName; per group write call_count = n and error_count = count where isError is true. (The events.tool_name promoted column exists if an SQL GROUP BY is preferred, but an in-memory group over readEvents is simplest and matches the single-read design.)

    No inline computation anywhere else — this function is the only writer of the three tables; it takes only `(db, runId)` and holds no session/timer/live-run state, so it cannot compute a metric inline by construction (TEL-02 / D-24).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted event log → projector | The projector reads only its own run's already-persisted, upstream-validated events via readEvents; no new external/untrusted input enters at this seam. |
| projector → projection tables | Writes folded rows to metrics/tool_calls/iterations in the same DB via parameterized prepared SQL. |

## STRIDE Threat Register

ASVS L1; block on any high-severity threat.

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-04-SQL | Tampering | projectMetrics DELETE/INSERT/SELECT | high | mitigate | Bound named params only (@run_id, @name, …); module-level static prepared-SQL constants mirroring evaluations.ts; runId is used solely as a bound parameter, never string-concatenated into SQL (T-1-SQL-01). |
| T-05-04-DOS | Denial of Service | single-pass fold over one run's log | low | accept | Fold is O(n) over one run's events, bounded by the run budget (maxTurns / maxWallClock). No new unbounded surface. |
| T-05-04-SC | Tampering | package installs (npm) | low | accept | No packages added — the projector uses only Node stdlib + the already-installed better-sqlite3 type. RESEARCH §Package Legitimacy Audit is empty by construction; no [ASSUMED]/[SUS] packages. |

**No new high-severity surface (justification):** the projector opens no network, spawns no process, and accepts no external input beyond a `runId` string consumed only as a bound SQL parameter.
