<objective>
Land the three additive, no-schema-change resolutions (D5-15 i/ii/iii) that the
orchestrator, summary, and HTML report all depend on but which no existing code
provides:

1. **`updateRunOutcome`** — the only writer of the terminal `runs` row. Today
   `persistManifest` inserts `status='pending'` and `updateRunComposite` touches
   only composite columns, so every row stays `'pending'` forever (research gap
   #3). Without this the summary/report/exit-code logic that keys off
   `runs.status` reads `'pending'` for every run.
2. **`linkExpectedScreenshot`** — persist the expected screenshot as a
   `screenshots role='expected'` artifact (mirroring `linkDiffScreenshot`, which
   only ever writes `role='diff'`) so `report <run_id>` regenerates the
   expected/generated/diff triptych self-contained from stored results alone
   (D5-06/D5-09), without re-reading the scenario spec.
3. **Production specs** — create `models/deepseek4pro.json` +
   `scenarios/dashboard/dashboard.yaml` (+ `expected.png`) from the proven
   `tests/fixtures/*` inputs so the D5-02 named flags resolve (only
   `stacks/angular.yaml` exists today; `models/` and `scenarios/` are absent).

Purpose: unblock CLI-01 (`run` writes a terminal, spec-resolvable row) and
REPORT-02 (`report` reads back a self-contained expected panel). This is the
foundational Wave-1 plan for Phase 5 — the orchestrator/projector/report plans
consume these three seams.

Output: two new storage functions in `src/storage/evaluations.ts`, three
production spec/data files, and two unit-test files proving each.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → storage fn (SQL) | Runtime values (`status`, `failedStage`, `finishedAt`, `viewport`) cross into DML — must stay parameterized, never interpolated. |
| bytes → filesystem | `expectedPng` bytes cross into `results/<runId>/expected.png` via `writeArtifact`; filename is a fixed literal, but the write must stay inside the run dir. |
| repo spec files → loaders | `models/*.json` + `scenarios/*/*.yaml` are trusted committed content, parsed by zod-validated loaders that throw on malformed input before any run starts (D-08). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Tampering | `updateRunOutcome` / `linkExpectedScreenshot` SQL | high | mitigate | Static SQL constants with named params only; zero string interpolation (mirrors T-1-SQL-01/02 and `updateRunComposite`/`linkDiffScreenshot`). No dynamic column/table names. |
| T-05-02 | Tampering | `expected.png` artifact write | medium | mitigate | Reuse `writeArtifact` — filename is the fixed literal `expected.png` and the existing V12 path-containment guard rejects any target escaping `results/<runId>/` before mkdir/write/insert. Not reimplemented. |
| T-05-03 | Information disclosure | `runs.status` surfaces `'pending'` briefly | low | accept | The row is `'pending'` only in the transient window between `persistManifest` and `updateRunOutcome`; the orchestrator always writes the terminal state. Not a security exposure — the projector/report tolerate partial rows (D5-05). |
| T-05-04 | Tampering | production spec files (`models/`, `scenarios/`) | low | accept | Committed, trusted repo content copied byte-for-byte from proven fixtures; loaded through zod-strict schemas that reject malformed specs at load (D-08). No external/user-supplied path in this plan. |
| T-05-SC | Tampering | npm/pip/cargo installs | low | accept | No package-manager installs in this plan — storage functions + data files only; nothing added to `package.json`. Package legitimacy gate N/A here. |
</threat_model>
