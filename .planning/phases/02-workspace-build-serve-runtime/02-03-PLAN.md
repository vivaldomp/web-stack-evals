<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stack.yaml command strings → execa spawn | Author-controlled today (Phase 2), will be agent-influenced by Phase 4 |
| Parent process env → child stage env | The only place secrets could leak into a spawned build/install process |
| `start` subprocess (and its descendants) → OS process table / port 4200 | Must not outlive the run |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-01 | Tampering | src/runtime/stage.ts runStage/startServer | high | mitigate | Array-form `execa(file, args, opts)` only, `shell` option never set — execa's default never spawns a shell, so no stack.yaml command string can achieve metacharacter injection. Acceptance criteria grep proves `shell: true` never appears in this file. |
| T-2-02 | Information Disclosure | src/runtime/env.ts buildAllowlistedEnv | high | mitigate | Default-deny allowlist (5 keys only) + `extendEnv: false` on every execa call in stage.ts — the parent process's full env (including any CI/npm tokens) is never visible to a spawned install/build/lint/test/start process; `--ignore-scripts` (declared in stacks/angular.yaml, Plan 02-02) additionally stops third-party postinstall hooks from running at all. |
| T-2-03 | Denial of Service (self-inflicted) | src/runtime/stage.ts startServer/killProcessTree | medium | mitigate | `detached: true` makes the start subprocess the leader of its own POSIX process group; `killProcessTree` signals the whole group via `process.kill(-pid, "SIGTERM")`, with `forceKillAfterDelay: 5000` as an escalation backstop — reaches grandchildren that `execa`'s own `cleanup` option alone would miss. |
| T-2-04 | Tampering | src/storage/storagePort.ts | medium | mitigate | The adapter delegates every write to Phase 1's already-path-contained `writeArtifact`/`appendEvent` verbatim — it changes only the id's string/number representation at the seam, never the containment logic itself. |
| T-2-05 | Denial of Service (self-inflicted) | src/runtime/stage.ts log capture | low | mitigate | `tailCap` caps the combined stdout+stderr log at 5MB, keeping the tail (where errors live) before it is ever written to disk by a later plan — a runaway stage cannot fill disk via its own log output. |
</threat_model>
