<objective>
Implement the driving Pi adapter: `src/agent/piAgentAdapter.ts`, the ONLY module that imports the Pi coding-agent SDK. It realises `AgentPort.runSession` (D4-13 live `AsyncIterable<AgentEventDraft>`) by narrowing the orchestrator's `AgentInput`, creating one cwd-locked Pi session with the resolved model + skills + in-memory auth, firing a single mockup-bearing prompt, and bridging Pi's `subscribe` event stream through the 04-04 `createEventMapper` into live-yielded canonical drafts. Pi's built-in retry is configured (not hand-rolled); a fatal agent failure ends the stream with a `benchmark_finished{agent_error}`.

Purpose: This is the seam that turns "drive a non-deterministic paid agent" into "consume a faithful, live event stream." Encapsulating Pi here structurally enforces AGENT-01 (ports-and-adapters, D-23): nothing else in the system knows Pi exists.
Output: `piAgentAdapter.ts` + a scriptable fake Pi session + a fake-driven adapter test + the AGENT-01 import-boundary guard — all typecheck-clean and green with zero network and zero cost.

Scope boundary: the three-ceiling budget loop (D4-01/D4-02), the `AbortController`/`session.abort()` ownership, process-tree teardown (D4-24), usage reconciliation on abort (D4-15), and the runtime path-containment guard (D4-23 full) are the SEPARATE next plan 04-06. This plan structures the run loop and retains the `session` handle so 04-06 slots those in without reshaping the generator.
